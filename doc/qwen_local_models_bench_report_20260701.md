## 1. 结论先行

如果只看服务器评测和通用 Agent 能力，**Qwen3-4B 是四个模型里最均衡的轻量模型**。但结合当前产品约束：模型要部署在炒菜机器人本机、硬件里不可能塞独立显卡、任务主要是从知识库检索菜谱后做少量个性化改写并输出 JSON，最终推荐需要调整为：

**端侧炒菜机器人默认选 Qwen3-1.7B，Qwen3-4B 作为高配机器人或边缘小主机备选，Qwen3-0.6B 只做路由/分类，Qwen3.6-27B 不进入机器人本机。**

原因很直接：炒菜机器人里的 Agent 不需要凭空创作复杂知识，也不应该让模型从长上下文里自由推理整套菜谱。正确做法是：**知识库精确检索菜谱，程序计算克数比例和安全边界，大模型只负责理解用户偏好、生成少量改动建议、输出严格 JSON**。在这个任务形态下，1.7B 的能力已经够用，而且 token 消耗、内存压力、CPU 推理延迟都更适合无显卡设备。

如果目标是“尽可能强的本地推理能力”，27B 仍然更稳，尤其在指令遵循、数学推理、复杂问题上更强。但它速度慢、显存和内存压力更大，不适合作为弱机器上的通用部署方案。

推荐排序：

1. **Qwen3-1.7B**：炒菜机器人端侧默认主模型。
2. **Qwen3-4B**：高配 CPU / 边缘小主机备选，质量更好但资源压力更高。
3. **Qwen3-0.6B**：不适合作为主 Agent，只适合分类、路由、意图识别、标题生成等辅助任务。
4. **Qwen3.6-27B**：服务器端高质量模型，适合离线评测、复核和数据生成，不适合迁移到机器人本机。

## 5. 单模型解读

### 5.1 Qwen3.6-27B

27B 的强项是复杂任务稳定性。它在 `instructfollow` 和 `reasonmath` 上明显领先，说明它更适合处理长链路推理、复杂约束、多步骤判断和高质量自然语言生成。

但它的代价也很明显：

- 生成速度只有 `33.71 tok/s`。
- Benchloop 总耗时 `2738s`，约 `45.6 分钟`。
- Value 分只有 `13.20`，说明单位成本下的收益不高。

适合用途：

- 服务器端高质量问答。
- 复杂健康建议生成前的二次复核。
- 高质量内容生成、复杂方案规划。
- 作为蒸馏、评测、裁判模型。

不建议用途：

- 弱机器部署。
- 高频、低延迟、多并发 Agent 主模型。
- 边缘端或普通办公电脑上的主模型。

### 5.2 Qwen3-4B

4B 是本次最值得关注的结果。它的 Overall `61.54`，只比 27B 低不到 1 分，但速度达到 `126.56 tok/s`。在工具调用上达到 `85.00`，数据抽取达到 `71.80`，这对 Agent 非常重要。

它的短板主要在：

- 复杂数学推理不如 27B。
- 指令遵循低于 27B。
- Coding 仍然只是可用，不是强项。

但对“烹饪 + 医疗大健康 Agent”来说，4B 的结构化输出、工具调用、数据抽取能力更关键。很多医疗大健康场景也不应该完全依赖模型裸推理，而应该通过 RAG、规则、禁忌表、药食冲突知识库和人工安全边界来增强。

适合用途：

- 主 Agent 基座。
- 菜谱生成、饮食计划、食材替换。
- 用户偏好理解。
- 健康目标下的膳食建议。
- 结构化输出 JSON。
- 调用外部知识库、营养数据库、禁忌检查工具。

建议定位：**默认首选模型**。

### 5.3 Qwen3-1.7B

1.7B 的速度很好，达到 `182.32 tok/s`，Toolcall 也有 `90.00`。这说明它可以承担一些轻量 Agent 入口任务，例如简单工具调用、意图识别、低复杂度问答。

但它在 Coding、Reason Math、Data Extract 上明显弱于 4B。作为主 Agent 时，遇到多约束菜谱、复杂疾病限制、药食冲突、多轮对话状态管理时，可靠性会明显下降。

适合用途：

- 低配机器上的轻量助手。
- 简单食谱问答。
- 菜名、食材、疾病标签抽取。
- 意图识别和任务路由。
- 辅助模型，负责把请求分发给 RAG 或规则系统。

不建议用途：

- 医疗大健康主 Agent。
- 复杂健康条件下的饮食计划生成。
- 需要严肃安全边界的最终回答模型。

建议定位：**低配备用模型 / 辅助 Agent 模型**。

### 5.4 Qwen3-0.6B

0.6B 的速度最高，`221.71 tok/s`，但质量分和可靠性过低。除了 speed 套件全部通过外，其他任务几乎不可用：Toolcall 只有 `1/15`，Coding、Data Extract、Instruct Follow、Reason Math 基本失败。

它不适合作为主 Agent，也不适合作为最终回答模型。

适合用途：

- 意图分类。
- 标题生成。
- 简单标签归类。
- 文本预处理。
- 路由模型，例如判断用户是在问“菜谱 / 营养 / 疾病禁忌 / 闲聊”。

建议定位：**辅助分类器，不作为主模型**。

## 6. 结合炒菜机器人约束后的选型建议

### 6.1 产品约束

当前 Agent 的真实运行环境和任务边界如下：

- 部署位置：炒菜机器人本机。
- 硬件限制：不放独立显卡，主要依赖 CPU / 集成 GPU / NPU 等低功耗硬件。
- 常见任务：用户说“我要吃辣椒炒肉”，系统从知识库拿到标准菜谱，再按肉和辣椒的克数、用户口味偏好做少量调整。
- 输出形式：先给用户展示修改后的菜谱，用户确认后输出一条结构化 JSON 给炒菜机器人执行。
- 关键要求：本地部署、低 token 消耗、低延迟、JSON 稳定、不要让模型自由编造菜谱。

这个约束会改变模型选择。原报告中 4B 的综合能力最均衡，但那是站在“有 4090D 服务器评测”和“通用 Agent 能力”的角度。真正放到炒菜机器人本机后，核心矛盾变成：**在无显卡、低 token、强结构化输出场景下，模型要足够小，同时不能牺牲基本指令理解和 JSON 输出能力**。

### 6.2 推荐架构

建议采用分层模型架构：

```text
用户输入
  -> 规则/小模型判断菜名、口味、份量、忌口
  -> 知识库精确检索标准菜谱
  -> 程序按食材克数计算比例和基础调料量
  -> Qwen3-1.7B 只做轻量个性化改写和 JSON 组装
  -> 用户确认
  -> 输出机器人可执行 JSON
```

其中：

- 菜谱知识不要塞进 prompt，而是从知识库检索得到。
- 调料比例优先由程序计算，模型只解释和微调。
- prompt 里只放“检索到的单道菜谱 + 用户偏好 + 输出 schema”，不要放整本知识库。
- 输出必须使用固定 JSON schema，并在程序侧校验。
- 医疗大健康相关约束应放在规则库和禁忌表里，模型不能单独决定安全边界。

### 6.3 最终推荐

| 场景 | 推荐模型 | 原因 |
|---|---|---|
| 炒菜机器人本机主模型 | Qwen3-1.7B | 速度快、资源压力小、工具调用表现好，适合 CPU-only 端侧部署 |
| 高配机器人 / 边缘小主机 | Qwen3-4B | 质量更稳，数据抽取更强，但需要更多内存和更高 CPU 算力 |
| 菜名识别 / 意图分类 / 路由 | Qwen3-0.6B | 极快，但不能承担主回答和 JSON 生成 |
| 服务器离线复核 / 数据生成 | Qwen3.6-27B | 复杂推理强，可用于生成训练数据、评测样本和人工复核 |

### 6.4 推荐部署形态

机器人本机推荐：

```text
Qwen3-1.7B Q4_K_M 或 Q5_K_M
context: 2048-4096
max output tokens: 512-1024
temperature: 0-0.2
top_p: 0.8-0.9
```

这样做的原因：

- 真实业务里单次只处理一道菜，不需要 32K 或 64K 大上下文。
- 菜谱从知识库来，不靠模型记忆。
- JSON 输出长度有限，通常几百 token 就够。
- 低温度可以提高结构化输出稳定性。
- 小上下文可以显著降低内存占用和推理延迟。

如果机器人硬件明显更强，例如有高性能 x86 CPU、16GB 以上内存、可接受更长响应时间，可以尝试：

```text
Qwen3-4B Q4_K_M
context: 4096-8192
max output tokens: 1024
temperature: 0-0.2
```

但不要一开始就上 4B。产品落地应该先验证 1.7B 在固定菜谱、固定 JSON schema、低 token prompt 下是否足够稳定。

### 6.5 Prompt 和 Token 控制建议

为了让端侧模型跑得动，prompt 应该尽可能短。推荐每次只给模型这些信息：

```json
{
  "user_request": "我要吃辣椒炒肉，少盐，多一点甜",
  "ingredients_actual": {
    "pork_g": 220,
    "pepper_g": 180
  },
  "retrieved_recipe": {
    "recipe_id": "chili_pork_001",
    "base_serving": "2人份",
    "ingredients": "...",
    "steps": "..."
  },
  "constraints": [
    "只允许基于 retrieved_recipe 修改",
    "必须输出固定 JSON schema",
    "不能新增机器人不支持的动作"
  ]
}
```

不要这样做：

- 不要把多个候选菜谱全塞进去。
- 不要把整套知识库塞进上下文。
- 不要让模型自己决定机器人动作名称。
- 不要让模型输出自然语言后再二次解析。

推荐输出形式：

```json
{
  "recipe_id": "chili_pork_001",
  "display_name": "少盐微甜辣椒炒肉",
  "user_confirmation_text": "已按你的口味减少盐、略微增加糖，是否开始烹饪？",
  "ingredients": [
    {"name": "猪肉", "amount_g": 220},
    {"name": "辣椒", "amount_g": 180}
  ],
  "seasonings": [
    {"name": "盐", "amount_g": 2.0},
    {"name": "糖", "amount_g": 3.0}
  ],
  "steps": [
    {"action": "heat_oil", "duration_s": 30},
    {"action": "stir_fry", "target": "pork", "duration_s": 90}
  ]
}
```

## 7. 性能和质量的取舍

从本次数据看，如果有服务器或较强边缘主机，4B 是最关键的能力拐点；但如果目标是无独立显卡的炒菜机器人本机，1.7B 才是更合理的落地起点。

| 对比 | 结论 |
|---|---|
| 4B vs 27B | 总分接近，4B 快约 3.75 倍，更适合服务器侧轻量 Agent |
| 1.7B vs 4B | 1.7B 更快、资源压力更小，更适合机器人端侧；但数据抽取和推理要靠规则/RAG兜底 |
| 0.6B vs 1.7B | 0.6B 只保留速度，主任务能力断崖式下降，只能做分类/路由 |

如果部署目标是炒菜机器人本机，优先级应该是：

1. 先尝试 Qwen3-1.7B Q4/Q5 量化。
2. 如果响应速度、内存和温度都能接受，再用 Qwen3-4B Q4 做对照测试。
3. 如果 1.7B 在 JSON 稳定性上仍然压力较大，就进一步缩短 prompt、强化 schema 校验，而不是直接换更小的 0.6B 当主模型。
4. Qwen3-0.6B 只作为辅助小模型，不做主回答和最终 JSON 生成。

## 8. 注意事项

Benchloop 的 `raw` harness 是通用能力评测，不是专门的烹饪或医疗大健康领域评测。因此本报告只能说明模型在速度、工具调用、数据抽取、指令遵循、基础推理等方面的相对表现。

对于垂直类 Agent，还需要补充领域评测集：

- 菜谱生成是否满足食材约束。
- 是否能正确处理过敏源。
- 是否能识别高风险医疗问题。
- 是否会给出危险的疾病治疗建议。
- 是否能根据糖尿病、高血压、高尿酸、肾病等条件调整饮食建议。
- 是否能稳定输出 JSON 或工具调用参数。

也就是说，Benchloop 可以帮我们做第一轮模型筛选，但不能替代最终业务验收。

## 9. 下一步建议

建议下一步建立一个面向炒菜机器人的小型垂直评测集，优先测试 Qwen3-1.7B，Qwen3-4B 作为质量上限对照。评测样本至少包含以下几类：

1. 菜谱检索后改写：辣椒炒肉、番茄炒蛋、青椒土豆丝等高频菜。
2. 食材克数变化：肉多、菜少、两人份变一人份、实际食材不足。
3. 口味偏好：少盐、多糖、少油、加辣、不放葱姜蒜。
4. 机器人动作约束：只能使用机器人支持的动作和锅具参数。
5. JSON schema 稳定性：字段完整、类型正确、单位统一、不能输出多余自然语言。
6. 用户确认流程：先输出确认文案，确认后才输出可执行步骤。
7. 安全边界：过敏、儿童、老人、慢病饮食提示只做饮食层面建议，不做诊断和治疗。

验收标准建议：

- Qwen3-1.7B 在固定 JSON schema 下通过率达到可接受水平，优先落地 1.7B。
- 如果 1.7B 主要问题是个别字段错误，优先通过程序校验、重试、schema 约束解决。
- 如果 1.7B 出现明显理解错误，再评估 Qwen3-4B 是否值得增加硬件成本。
- 27B 不作为机器人端侧部署目标，只作为离线评测、数据生成、复杂样本复核使用。
